# 知识库同步与文档更新规则

## 一、文档更新触发机制

### 1.1 自动触发场景

| 场景 | 触发条件 | 更新文档类型 |
|:--|:--|:--|
| **代码变更** | 提交代码后 | 开发记录、代码修改清单 |
| **配置变更** | 修改sdkconfig后 | 技术文档、配置说明 |
| **Bug修复** | 完成问题调试后 | 调试记录、经验总结 |
| **功能完成** | 功能开发完成后 | 开发记录、需求执行清单 |
| **架构调整** | 重构或调整架构后 | 架构文档、技术文档 |

### 1.2 手动触发命令

| 命令 | 说明 |
|:--|:--|
| `/doc-update-workflow` | 执行完整文档更新工作流 |
| `更新开发记录` | 更新当前开发记录 |
| `同步知识库` | 将知识库同步到项目文档 |

---

## 二、文档更新检查清单

### 2.1 代码变更后检查

每次提交代码后，检查以下项目：

- [ ] 开发记录是否记录了代码变更
- [ ] 新增文件是否添加到文档索引
- [ ] 删除文件是否从文档中移除
- [ ] API变更是否同步到接口文档
- [ ] 是否有值得记录的开发经验

### 2.2 配置变更后检查

修改sdkconfig或其他配置文件后：

- [ ] 配置项变更是否记录
- [ ] 配置项的用途是否说明
- [ ] 相关联的技术文档是否更新

### 2.3 问题解决后检查

完成问题调试后：

- [ ] 调试记录是否完整
- [ ] 根因分析是否准确
- [ ] 解决方案是否记录
- [ ] 经验总结是否提炼

## 三、知识库同步规则

### 3.1 知识库创建时机

| 场景 | 知识库内容 | 同步目标 |
|:--|:--|:--|
| 解决关键问题 | 根因+解决方案 | 调试记录 |
| 确定技术方案 | 方案要点+决策依据 | 技术文档 |
| 完成模块开发 | 模块架构+核心API | 开发记录 |
| 发现重要配置 | 配置项+注意事项 | 技术文档 |
| 积累开发经验 | 经验教训+最佳实践 | 开发记录 |

### 3.2 知识库同步到文档

**触发条件**：
1. 同一主题知识库记录累计 ≥3 条
2. 用户明确要求同步
3. 任务阶段结束

**同步步骤**：
1. 检索相关知识库记录
2. 按主题分类整理
3. 生成/更新对应文档
4. 标记已同步的记录

### 3.3 文档同步到知识库

**场景**：长文档的关键结论需要快速检索

**操作**：
```
提取文档关键信息 → 创建知识库记录
标签: [模块名, 技术点, 日期]
```

## 八、文档质量检查

### 8.1 内容准确性

| 检查项 | 方法 |
|:--|:--|
| 技术参数 | 与源代码对比 |
| API接口 | 与代码签名对比 |
| 引用链接 | 访问测试 |
| 格式规范 | 对照模板 |

### 8.2 完整性检查

| 检查项 | 标准 |
|:--|:--|
| 章节完整性 | 所有模板章节齐全 |
| 代码示例 | 核心功能有示例 |
| 注意事项 | 包含可能的坑 |
| 参考资料 | 包含相关链接 |

### 8.3 可读性检查

| 检查项 | 标准 |
|:--|:--|
| 结构清晰 | 层次分明 |
| 图表辅助 | 适当使用Mermaid |
| 颜色标注 | 关键信息突出 |
| 表格整理 | 数据结构化 |
